## Decisions & rationale (read these)

1. **Motion CSS went into `site.css`, not `dissensus.css`.** The brief said
   append the motion block to `dissensus.css` "the shared design system." But
   **no page links `dissensus.css`** ‚Äî it's a stale 52 KB monolith (Jun 27) that
   was split into `system.css` (tokens+base) + `site.css` (components, Jul 2),
   which is what every page actually loads. Appending to the dead file would
   have styled nothing. I put the motion layer in `site.css` so every page truly
   inherits it, which is the brief's intent. `dissensus.css` remains untouched
   and unreferenced (candidate for deletion later).

2. **Homepage components scoped under `main.home`.** The prototype reuses class
   names (`.hero`, `.card`, `.btn`, `.stats`, `.eq`, `.block`, `.kicker`,
   `.dot`) that mean *different things* in `system.css`/`site.css`. Scoping the
   ported CSS under `.home` avoids all collisions without renaming, and keeps
   the shared nav/footer on their shared classes.

3. **The homepage is theme-adaptive (not dark-locked like the prototype).**
   Every colour maps to the shared tokens, so the page flips correctly with the
   site's existing light/dark toggle (which lives in the nav). The prototype was
   dark-only; making it token-driven keeps the homepage consistent with the rest
   of the site and the toggle working. The decorative hero canvases are tuned for
   the dark default; on light theme they render as subtle wine-on-cream ‚Äî fine,
   but that's the one spot that's visibly dark-first.

4. **Nav mark is theme-swapped.** The brief said use
   `assets/dissensus-mark-mono-white.png` in the nav. A white-only mark is
   invisible on the light cream background, so I ship **both** the white mark
   (dark theme) and the wine mark (light theme) and swap via CSS. Still the real
   PNG asset the brief specified ‚Äî just visible in both themes.

5. **Favicons served from root (`/favicon.ico`, ‚Ä¶).** Matches the exact link
   paths in the brief. The old `assets/favicon-{32,64}.png` /
   `assets/apple-touch-icon.png` references were replaced (not doubled up) so the
   browser doesn't pick a stale icon. Those old files still exist in
   `public/assets/` but are now unreferenced (cleanup candidate).

6. **OG image policy.** Shipped `og-image.png` consistently across all 12 wired
   pages: updated the two pages that already had `og:image` (`about`,
   `services`) **in place** (no duplicate tags), and added it to the pages that
   had none. `research.html`/`tools.html` still point at the old `logo.png`
   because they're generated (phase-2). The site-wide JSON-LD `logo` field still
   points at `assets/logo.png` (left as-is; it's a logo, not a social card).

Cloudflare Pages auto-deploys `public/` on push to `master`. **I did not commit
or push.** Current git state:

  June design rehaul is already on origin. A bare `git push` right now ships
  **nothing** ‚Äî everything new is uncommitted in the working tree.

- The working tree contains **two unrelated things**:
  1. **This redesign** ‚Äî the `public/**` changes + new assets above.
  2. **A pre-existing `M papers.json`** ‚Äî modified *before* this session, **not

     `public/`, so it isn't served directly ‚Äî but a blanket
     `git add -A && commit && push` would **bundle it into the same commit** as
     the redesign. Review/stage it separately if you don't want it riding along.

  `logo-assets/`, `WEBSITE_PLAN_JUL2026.md`, this file) would also be committed
  by `git add -A`. None are under `public/`, so **none deploy** ‚Äî but decide
  whether you want them in the repo or gitignored.

**Bottom line:** a push (after committing) deploys the homepage redesign +
new brand assets live, AND folds in whatever else you stage ‚Äî including that
unrelated `papers.json`. Review before committing.
